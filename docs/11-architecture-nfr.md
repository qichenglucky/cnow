# 架构设计与实现约束

## 1. 目标
本文件定义即码 CodeNow.ai 在架构与实现层必须满足的非功能性要求，覆盖：
- 可用性与可靠性
- 性能与扩展性
- 可维护性与扩展性
- 安全性
- 可观测性
- 高效交付与部署
- 成本控制

## 2. 可用性与可靠性

### 2.1 底线原则
- 系统可以慢，但不能挂
- 核心链路必须具备冗余、降级和故障转移能力
- 任何单点都必须被识别并消除

### 2.2 SLO / SLA 建议
- 平台管理面可用性目标：99.99%
- 核心发布接口可用性目标：99.95%
- 非核心 AI 辅助能力可用性目标：99.5%

### 2.3 关键冗余设计
- API Gateway 多实例部署
- Web/BFF 无状态化，多副本
- PostgreSQL 主备或高可用集群
- Redis Sentinel / Cluster
- Kafka 多副本与 ISR 校验
- Temporal 高可用部署
- 对象存储高可用或云厂商托管

### 2.4 故障转移与降级
- 核心读接口支持降级为缓存/快照数据
- AI 能力不可用时，创建流程退化为纯表单模式
- 监控或日志外部系统异常时，保留平台内部摘要与最近快照
- 外部系统故障时，工作流进入可恢复等待态，不直接丢失上下文

### 2.5 一致性策略
- 业务主状态使用强一致事务
- 外部系统接入使用最终一致性
- 发布、回滚、审批等关键动作通过工作流和事件表兜底
- 所有副作用操作必须具备幂等键

### 2.6 RPO / RTO
- 核心业务数据 RPO 目标：0 到 5 分钟
- 核心业务恢复 RTO 目标：15 分钟内恢复管理面与发布主链路
- 外部观测系统恢复可晚于平台主链路，但不能丢失平台内审计事件

## 3. 性能与可扩展性

### 3.1 性能基线
- 核心查询接口 P99 < 200ms
- 核心写接口 P99 < 500ms
- 创建服务和发起发布的同步返回应尽量小于 1 秒
- 长流程动作必须异步化，不阻塞用户主线程

### 3.2 容量与伸缩
- Web/API 层必须无状态，可水平扩展
- Session、草案、文件、截图、日志摘要等状态剥离到 Redis / 对象存储 / 数据库
- 任务执行层支持 worker 横向扩容
- 发布和创建流程的峰值压力通过队列和工作流吸收

### 3.3 容量规划建议
- 数据库连接池按实例和流量预留 2 到 3 倍峰值缓冲
- Kafka 消息堆积设置报警阈值和扩容阈值
- Temporal 队列长度超过阈值时自动限流
- API 层超过压力阈值后返回明确的限流提示，而不是硬崩溃

### 3.4 软着陆机制
- 限流
- 排队
- 降级
- 只读模式
- 异步化

## 4. 可维护性与扩展性

### 4.1 结构原则
- 遵循分层架构
- 领域边界清晰
- 核心流程模块化
- 外部系统适配器化
- AI 能力独立成模块

### 4.2 推荐分层
- Interface 层：HTTP、BFF、RPC
- Application 层：用例编排、事务控制
- Domain 层：领域模型、规则、状态机
- Infrastructure 层：数据库、消息、外部系统适配

### 4.3 开闭原则
- 核心流程通过策略模式、适配器模式、SPI 扩展
- 新增发布策略、新增外部系统、新增 AI 能力时，优先扩展，不改核心链路

### 4.4 技术债管理
- 允许临时硬编码，但必须写入 ADR
- 每个 ADR 必须注明：
  - 决策原因
  - 影响范围
  - 还债时间
  - 替代方案
- 超过一个迭代未偿还的临时方案必须重新评审

## 5. 安全性

### 5.1 纵深防御
- 传输加密：TLS / mTLS
- 存储加密：数据库、对象存储、密钥管理
- 应用层防护：SQL 注入、XSS、CSRF、SSRF、命令注入防护

### 5.2 最小权限原则
- 服务间调用使用最小权限凭证
- 数据库账号按服务拆分权限
- 运维账号不使用万能账号
- AI 服务只能访问授权上下文

### 5.3 敏感数据处理
- 日志中不得打印密码、Token、证书私钥、身份证号等敏感信息
- 所有敏感字段默认脱敏
- 满足《个人信息保护法》及内部合规要求

### 5.4 审计安全
- 生产发布、回滚、审批、域名变更必须留痕
- 审计记录不可篡改
- 审计查询必须具备权限控制

## 6. 可观测性

### 6.1 日志
- 全链路 TraceId 透传
- RequestId 与 TraceId 双标识
- 统一结构化日志格式
- 关键动作都要有审计日志和运行日志

### 6.2 指标
- 黄金指标：
  - 延迟
  - 流量
  - 错误
  - 饱和度
- 关键系统指标：
  - 工作流积压
  - 数据库连接池
  - Kafka 堆积
  - API 5xx 比例
  - 发布失败率

### 6.3 告警
- 核心链路异常要先于用户感知
- 告警必须关联服务、环境、版本、发布单
- 告警必须有可操作建议，不只是一条消息

### 6.4 故障演练
- 支持混沌工程和故障注入
- 定期验证：
  - DB 故障
  - Redis 故障
  - Kafka 堆积
  - 外部日志系统不可用
  - AI 服务不可用

## 7. 高效交付与部署

### 7.1 环境一致性
- 开发、测试、预发、生产尽量容器化一致
- 所有环境部署目标统一到 K8s
- 配置通过环境变量和配置中心注入

### 7.2 发布策略
- 支持灰度发布
- 支持金丝雀发布
- 支持快速回滚
- 回滚目标应在 1 分钟内完成切换或进入回滚执行态

### 7.3 数据库变更
- DDL 必须向前兼容
- 优先加字段，不删字段
- 先扩展后收敛
- 数据迁移必须可回滚或可补偿

## 8. 成本控制

### 8.1 资源利用率
- 平台核心应用必须支持弹性扩缩容
- 空闲 worker 必须可缩容
- 非核心观测能力允许按需调度

### 8.2 技术复杂度控制
- 默认不引入 Service Mesh 作为首期前提
- 只在明确收益大于学习/运维成本时引入新中间件
- 新技术引入必须通过 ADR 和收益评估

## 9. 架构验收清单
- 是否消除了核心单点故障
- 是否定义并实现 SLO / RPO / RTO
- 是否满足核心接口性能目标
- 是否支持水平扩展与软着陆
- 是否有清晰分层与模块边界
- 是否有 AI 边界与权限控制
- 是否有全链路日志、指标、告警
- 是否具备发布灰度、回滚和故障演练能力
- 是否满足成本控制要求
